# Engagement Schema

> 本文档定义"一份合格的读者参与度规则文档"的结构规范。
> 读者参与度 = 钩子机制 + 回报机制 + 爽感/满足感模式。
> 这些具体内容由理解层根据叙事结构生成，本 schema 只定义框架。

---

## 文档结构

### 1. 钩子类型定义（Hook Types）

定义本项目使用的钩子类型。类型名称和数量不固定。

```markdown
## 钩子类型

| 类型 | 定义 | 强度判定标准 | 适用场景 |
|------|------|-------------|---------|
| crisis | 危机逼近，读者担心角色安全 | 强：生死攸关；中：利益威胁；弱：潜在风险 | 战斗前、阴谋暴露前 |
| mystery | 信息缺口，读者想知道答案 | 强：核心谜题；中：支线疑问；弱：细节好奇 | 发现异常、线索出现 |
| ... | ... | ... | ... |
```

### 2. 回报类型定义（Payoff Types）

定义本项目使用的微回报/满足感类型。

```markdown
## 回报类型

| 类型 | 定义 | 满足感来源 |
|------|------|-----------|
| info_reveal | 读者/角色获知新信息 | 好奇心满足 |
| power_display | 角色展示实力 | 代入感爽快 |
| pattern_recognition | 读者发现隐藏模式 | 智力优越感 |
| ... | ... | ... |
```

### 3. 爽感模式定义（Satisfaction Patterns）（可选）

如果项目有特定的爽感模式（如传统网文的8种模式），在此定义。
如果项目不适用"爽感"概念（如文学小说），此 section 可省略或替换为其他满足感模式。

```markdown
## 爽感模式

### 模式名称
**结构**: 铺垫要素 → 触发条件 → 交付方式 → 余韵处理
**节奏**: 铺垫X% → 交付Y% → 余韵Z%
**质量标准**: ...
**适用场景**: ...
```

### 4. 密度要求（Density Requirements）

```markdown
## 密度要求

| 维度 | 频率 | 检测窗口 |
|------|------|---------|
| 钩子 | 每章至少1个（弱及以上） | 单章 |
| 微回报 | 每章至少1个 | 单章 |
| 爽感模式 | 每5章至少1个完整交付 | 5章滚动 |
| 组合高潮 | 每10-15章至少1个 | 15章滚动 |
```

### 5. 多样性要求（Diversity Requirements）

```markdown
## 多样性要求

| 维度 | 规则 |
|------|------|
| 钩子类型 | 不得连续3章使用相同类型 |
| 回报类型 | 单一类型不超过滚动窗口的60% |
| 爽感模式 | 单一模式不超过总量的50% |
```

### 6. 分篇差异化（Arc Variations）（可选）

不同篇章可以有完全不同的参与度规则。

```markdown
## 分篇差异化

### A篇章
钩子偏好: crisis, desire
爽感模式: 信息差碾压, 资源争夺胜利
特殊规则: 每个爽点必须同时推进A的悲剧深度

### D篇章
钩子偏好: mystery, choice
爽感模式: 静水识破, 暗线布置
特殊规则: D的行动不能超出D当前知识边界
```

## 转化为 Checker

engagement-schema 定义的是内容框架。
实际检测时，理解层会将此框架填充为一份符合 checker-schema 的完整检测文档。

转化规则：
- 密度要求中的"每章至少1个" → 硬约束
- 多样性要求 → 软约束
- 分篇差异化中的特殊规则 → 作用域限定的约束

---

## Schema 规则

1. 钩子类型至少定义 2 种
2. 回报类型至少定义 2 种
3. 爽感模式是可选的（不是所有叙事都有传统"爽感"）
4. 密度要求必须定义（即使阈值很低）
5. 如果项目有多篇章且风格不同，必须定义分篇差异化
6. 所有类型名称是自由文本，不限于预定义列表
